# Synthesis, Build, And AFI Creation

This document covers the AWS HDK environment, quick synthesis-oriented checks, BuildAll, and converting a completed build into an AFI. For simulation and debug workflows, see `Verification.md`. For using an already-created AFI on an F2 instance, see `FPGA.md`. For HDK reference material (clocks, shell spec, AFI lifecycle), see [HDK.md](HDK.md).

## Initialize HDK Environment

Before any AWS build step, the HDK environment variables must be populated.

> **Warning**: `hdk_setup.sh` cannot be sourced from this project root. It calls `git rev-parse --show-toplevel`; since `SatSwarmV2` is not a git repository, that leaves `AWS_FPGA_REPO_DIR` empty and breaks downstream paths.

Set all required vars manually:

```bash
export AWS_FPGA_REPO_DIR=/home/ubuntu/src/project_data/SatSwarmV2/src/aws-fpga
export HDK_DIR=$AWS_FPGA_REPO_DIR/hdk
export HDK_COMMON_DIR=$HDK_DIR/common
export HDK_SHELL_DIR=$HDK_COMMON_DIR/shell_stable
export HDK_SHELL_DESIGN_DIR=$HDK_SHELL_DIR/design
export HDK_IP_SRC_DIR=$HDK_COMMON_DIR/ip/cl_ip/cl_ip.srcs/sources_1/ip
export HDK_BD_SRC_DIR=$HDK_COMMON_DIR/ip/cl_ip/cl_ip.srcs/sources_1/bd
export HDK_BD_GEN_DIR=$HDK_COMMON_DIR/ip/cl_ip/cl_ip.gen/sources_1/bd
export CL_DIR=$AWS_FPGA_REPO_DIR/hdk/cl/examples/cl_satswarm
export FAAS_CL_DIR=$CL_DIR
export VIVADO_TOOL_VERSION=2025.2
export XILINX_VIVADO=/opt/Xilinx/2025.2/Vivado
export PATH=/opt/Xilinx/2025.2/Vivado/bin:$PATH
```

These exports must live in the same shell as the build command that follows.

## Quick-Check Workflow

Do the cheap checks before committing to a long Vivado run.

### Simulation-first checks

Use `Verification.md` for the simulation gate:

- `run_bigger_ladder.sh` for correctness
- XSim bridge smoke / full AWS shell regression when shell wiring changed

For shell interface constraints (4K boundary, 8 µs timeout, 64B AxSIZE), see [AWS_Shell_ERRATA.md](../src/aws-fpga/hdk/docs/AWS_Shell_ERRATA.md).

### Synthesis-oriented checks

Before BuildAll, use the AWS tools to catch obvious synthesis issues:

1. **Syntax / lint**
  Compile the RTL under `$CL_DIR/design/` with `xvlog -lint +all` or equivalent.
2. **Out-of-context synthesis**
  Use `synth_design -mode out_of_context` on the top module to catch elaboration failures early.
3. **DRC sanity**
  Run `report_drc` on the synthesized design.
4. **Constraint sanity**
  Confirm clocks / XDC expectations match the CL being built.
5. **BRAM inference spot checks**
  If `pse.sv` or `trail_manager.sv` changed, run:

```bash
cd /home/ubuntu/src/project_data/SatSwarmV2
bash deploy/check_inference.sh pse
```

The fast rule of thumb is simple: if a change touches algorithmic behavior, start in `Verification.md`; if it touches inference, clocks, wrapper wiring, or build plumbing, come here before running BuildAll.

## Running SatSwarm Synthesis

Once simulation and inference are clean, kick off synthesis.

```bash
cd /home/ubuntu/src/project_data/SatSwarmV2/deploy
./run_synthesis.sh 2>&1 | tee logs/synth_explore.log &
```

### Current launcher settings

- `--aws_clk_gen` is required when using custom clock recipes.
- `A2` is the current recipe. `clk_main_a0` is always 250 MHz regardless of A-recipe; the A2 `gen_clk_extra_a1` (15.625 MHz) is unused by CL logic.
- `A1 (150 MHz)` still requires additional RTL pipelining work.

See [Clock_Recipes_User_Guide.md](../src/aws-fpga/hdk/docs/Clock_Recipes_User_Guide.md) for the full recipe table and [AWS_CLK_GEN_spec.md](../src/aws-fpga/hdk/docs/AWS_CLK_GEN_spec.md) for the clock generator IP.

### Clock-domain note

The wrapper crosses shell-facing OCL, PCIS, and DDR paths through explicit AXI CDC blocks. The solver (behind `satswarm_core_bridge` / Mega RTL) runs on `clk_solver` (15.625 MHz, 64 ns period) generated by a **CL-owned MMCME4_ADV** (`u_mmcm_solver`) from `clk_main_a0` (250 MHz). Shell-facing logic remains on `clk_main_a0`. Vivado auto-propagates a `create_generated_clock` through the MMCM; `cl_timing_user.xdc` provides `set_false_path` exceptions for the CDC crossings.

`aws_clk_gen` is instantiated with all clock groups disabled (`CLK_GRP_A_EN=0`) — it has no active MMCMs. See [§ clk_solver constraint history](#clk_solver-constraint-history) for the evolution from fabric divider to CL-owned MMCM.

## Running Full Build (BuildAll)

Once synthesis writes a clean checkpoint (`*.synthesis.dcp`), run implementation and tar generation.

```bash
export AWS_FPGA_REPO_DIR=/home/ubuntu/src/project_data/SatSwarmV2/src/aws-fpga
export HDK_DIR=$AWS_FPGA_REPO_DIR/hdk
export HDK_COMMON_DIR=$HDK_DIR/common
export HDK_SHELL_DIR=$HDK_COMMON_DIR/shell_stable
export HDK_SHELL_DESIGN_DIR=$HDK_SHELL_DIR/design
export HDK_IP_SRC_DIR=$HDK_COMMON_DIR/ip/cl_ip/cl_ip.srcs/sources_1/ip
export HDK_BD_SRC_DIR=$HDK_COMMON_DIR/ip/cl_ip/cl_ip.srcs/sources_1/bd
export HDK_BD_GEN_DIR=$HDK_COMMON_DIR/ip/cl_ip/cl_ip.gen/sources_1/bd
export CL_DIR=$AWS_FPGA_REPO_DIR/hdk/cl/examples/cl_satswarm
export FAAS_CL_DIR=$CL_DIR
export VIVADO_TOOL_VERSION=2025.2
export XILINX_VIVADO=/opt/Xilinx/2025.2/Vivado
export PATH=/opt/Xilinx/2025.2/Vivado/bin:$PATH
LOG=/home/ubuntu/buildall_$(date +%Y%m%d_%H%M%S).log
cd $CL_DIR/build/scripts
nohup python3 aws_build_dcp_from_cl.py \
  --cl cl_satswarm --aws_clk_gen \
  --clock_recipe_a A2 --clock_recipe_b B0 --clock_recipe_c C0 \
  > "$LOG" 2>&1 &
echo "PID=$! LOG=$LOG"
```

> **Important**: run the build script from `$CL_DIR/build/scripts/`. Invoking it from another directory with a full path can break relative TCL includes.

The output tar lands at:

`$CL_DIR/build/checkpoints/<tag>.Developer_CL.tar`

Older notes referring to a `to_aws/` subdirectory are stale.

### Automated AFI Submission In Grid Sweeps

`deploy/run_grid_sharing_builds.sh` now auto-submits AFIs for each successful run by default.

After a successful BuildAll run with a discovered `latest_tar`, the script:

1. uploads the tar to S3 (`s3://$AFI_S3_BUCKET/$AFI_S3_DCP_PREFIX/<tag>.Developer_CL.tar`)
2. calls `aws ec2 create-fpga-image`
3. writes the create response JSON to `$RUN_DIR/afi_create_<run_label>_<stamp>.json`
4. appends AFI metadata to summary CSV columns:
  - `afi_status`
  - `afi_id`
  - `agfi_id`
  - `afi_json`

Default environment settings used by the script:

```bash
AUTO_CREATE_AFI=1
AWS_REGION=us-east-1
AFI_S3_BUCKET=satswarm-v2-afi-624824941978
AFI_S3_DCP_PREFIX=dcp
AFI_S3_LOGS_PREFIX=logs
```

To disable AFI submission for a specific run:

```bash
AUTO_CREATE_AFI=0 bash deploy/run_grid_sharing_builds.sh
```

### Skipping synthesis (`ImplCL`)

If `post_synth.dcp` is already clean, you can skip synthesis:

```bash
cd $CL_DIR/build/scripts
python3 aws_build_dcp_from_cl.py \
  --cl cl_satswarm -f ImplCL -t "2026_03_18-163435" \
  --aws_clk_gen --clock_recipe_a A2 --clock_recipe_b B0 --clock_recipe_c C0 \
  > /home/ubuntu/build_impl.log 2>&1 &
```

`ImplCL` skips tarball creation, so you need a follow-on tar creation step if you use that flow.

## Fast Diagnostic Builds

### Reading the critical path without waiting

Timing reports are written at each checkpoint during a full build. Once placement finishes (roughly 30–35 min into a build), `post_place_timing.rpt` is ready. You do not need to wait for phys_opt or route to see the worst path:

```bash
# After placement finishes:
head -270 $CL_DIR/build/reports/cl_satswarm.<tag>.post_place_timing.rpt
```

The first violation block shows: Slack, Source FF, Destination cell, data path delay breakdown (logic vs. routing), and the full cell-by-cell path.

### Using Default directives to halve runtime

The current BuildAll uses `SSI_SpreadLogic_high` (placement), `AggressiveExplore` (phys_opt), and `AggressiveExplore` (route). These are the most expensive strategies. To get a WNS reading faster — for RTL iteration, not for a submission build — use `Default` for all three:

```bash
cd $CL_DIR/build/scripts
python3 aws_build_dcp_from_cl.py \
  --cl cl_satswarm --aws_clk_gen \
  --clock_recipe_a A2 --clock_recipe_b B0 --clock_recipe_c C0 \
  --no-encrypt \
  -p Default \
  -o Default \
  -r Default \
  > /home/ubuntu/build_fast_$(date +%Y%m%d_%H%M%S).log 2>&1 &
```

Typical wall-clock comparison on this 8-core machine:

| Strategy | Synth | Place | PhysOpt | Route | Total |
|---|---|---|---|---|---|
| Aggressive (current) | ~20 min | ~10 min | ~45 min | ~20 min | ~95 min |
| Default | ~20 min | ~5 min | ~5 min | ~10 min | ~40 min |

The WNS from a `Default` run is pessimistic relative to the aggressive run but is a valid lower bound. If `Default` shows WNS > 0 the aggressive build will also pass; if `Default` shows a large negative WNS, the RTL needs pipelining regardless of strategy.

### clk_solver constraint history

**The original broken constraint and why it failed.**

The original `cl_timing_user.xdc` used:

```tcl
create_generated_clock \
    -name   clk_solver \
    -source [get_pins u_solver_clk_div/u_bufg/I] \
    -divide_by 16 \
    [get_pins u_solver_clk_div/u_bufg/O]
```

`create_generated_clock` requires the `-source` pin to lie on an existing clock network (a clock port, MMCM output, or BUFGCE output). The BUFG input here (`u_solver_clk_div/u_bufg/I`) is driven by `div_pre` — the Q output of a counter flip-flop clocked by `clk_main_a0`. That is a data/fabric signal. Vivado does not establish the intended generated-clock relationship from this source; instead, downstream logic is analyzed under the propagated parent clock (`clk_main_a0`, 4 ns period). All solver FFs in `u_cae` and `u_pse` were placed in path group `clk_main_a0` and analyzed at 250 MHz.

The result: the worst path (40 logic levels, 10.618 ns data path delay from `u_cae/trail_read_var_r_reg[1]` → `u_pse/u_prop_q_fifo/mem_reg_384_447_14_20/WE`) was being evaluated against a 4 ns budget. WNS was -6.818 ns post-place, and no placement or routing strategy can close a 10 ns path into 4 ns — phys_opt with `AggressiveExplore` recovered only 0.128 ns. This was confirmed in build `2026_03_19-020552` `post_place_timing.rpt` (Requirement: 4.000 ns, both source and destination FFs stamped `clk_main_a0 period=4.000ns`).

If the solver had been truly unconstrained (e.g., as in the earlier `gen_clk_extra_a1`-based builds described in `HANDOFF.md`), Vivado would not have performed meaningful timing analysis on those paths at all — not "infinite slack," but not analyzed. The current failure was caused by active but incorrect analysis under the wrong clock.

**Intermediate fix (now superseded).**

The fabric `solver_clk_div` constraint was replaced with `create_clock -period 64.000 -name clk_solver [get_pins u_solver_clk_div/u_bufg/O]`. This declared `clk_solver` as an independent primary clock, which was analytically safe because all CDC crossings had `set_false_path` exceptions. The build (tag `2026_03_19-031457`) closed timing locally (WNS=+0.711 ns) but the AFI failed with REQP-123 because `aws_clk_gen` still had `CLK_GRP_A_EN(1)` with `.i_clk_hbm_ref(1'b0)` — the Group A MMCM had a dead clock.

**Current fix: CL-owned MMCM (tag `2026_03_19-051231`).**

The fabric divider was replaced with a standalone MMCME4_ADV (`u_mmcm_solver`) in `cl_satswarm.sv`:
- Input: `clk_main_a0` (250 MHz), CLKFBOUT_MULT_F=5, CLKOUT0_DIVIDE_F=80 → 15.625 MHz
- Output: BUFG (`u_bufg_solver`) → `clk_solver`
- Solver reset gated on `mmcm_solver_locked`
- `CLK_GRP_A_EN(0)` in `aws_clk_gen` — no Group A MMCM, no REQP-123
- Vivado auto-propagates `create_generated_clock` through the MMCM; no manual constraint needed
- `cl_timing_user.xdc` has `set_false_path` referencing `[get_clocks -of_objects [get_pins u_mmcm_solver/CLKOUT0]]`

**Verification steps after any build.**

```tcl
report_clocks
```
Confirm the MMCM-derived solver clock appears with period ~64 ns.

```tcl
check_timing
```
Confirm no solver-domain endpoints appear in the "unconstrained" section.

```tcl
report_clock_interaction
```
Confirm the `clk_main_a0` → solver-clock and solver-clock → `clk_main_a0` entries are marked as false paths.

## Convert Build Output To An AFI

After BuildAll completes, upload the tarball to S3 and request AFI creation.

Tarball location:

`$CL_DIR/build/checkpoints/<timestamp>.Developer_CL.tar`

### 1. Upload to S3

Bucket:

`satswarm-v2-afi-624824941978`

```bash
aws s3 cp \
  $CL_DIR/build/checkpoints/<tag>.Developer_CL.tar \
  s3://satswarm-v2-afi-624824941978/dcp/<tag>.Developer_CL.tar
```

### 2. Create the AFI

```bash
aws ec2 create-fpga-image \
  --region us-east-1 \
  --name "SatSwarmV2-<grid>" \
  --description "SatSwarm V2 CDCL solver, <NxN> grid, A2 build" \
  --input-storage-location Bucket=satswarm-v2-afi-624824941978,Key=dcp/<tag>.Developer_CL.tar \
  --logs-storage-location Bucket=satswarm-v2-afi-624824941978,Key=logs/
```

Save both:

- `FpgaImageId` (`afi-*`) for polling
- `FpgaImageGlobalId` (`agfi-*`) for loading

### 3. Poll for availability

```bash
aws ec2 describe-fpga-images \
  --fpga-image-ids <afi-id> \
  --query 'FpgaImages[*].{Id:FpgaImageId,State:State}' \
  --region us-east-1
```

When the state becomes `available`, the AFI is ready to load on F2.

For the actual on-instance load / run workflow after the AFI exists, switch to `FPGA.md`. For AFI lifecycle and manifest details, see [Amazon_FPGA_Images_Afis_Guide.md](../src/aws-fpga/hdk/docs/Amazon_FPGA_Images_Afis_Guide.md) and [AFI_Manifest.md](../src/aws-fpga/hdk/docs/AFI_Manifest.md).

## Known Warnings

These warnings have historically been non-blocking:

| Code                  | Cause                                                                                                           | Impact                                          |
| --------------------- | --------------------------------------------------------------------------------------------------------------- | ----------------------------------------------- |
| `Designutils 20-1280` | Clock IPs registered as RTL sources instead of IP blocks, so corresponding XDCs do not apply as IP constraints. | None for CL logic.                              |
| `Synth 8-6858/6859`   | Multi-driven payload outputs on unused mesh edges.                                                              | None if those edges are intentionally tied off. |

**Resolved timing issue**: Build `2026_03_19-020552` failed timing (WNS=-6.66) due to incorrect `create_generated_clock`. The fabric divider was subsequently replaced with a CL-owned MMCME4_ADV (tag `2026_03_19-051231`, WNS=+0.711 ns). See [§ clk_solver constraint history](#clk_solver-constraint-history).

**Resolved REQP-123**: AFI `afi-064b74577e3b2f258` (fabric divider) failed AWS bitgen with DRC REQP-123 because `CLK_GRP_A_EN(1)` + `.i_clk_hbm_ref(1'b0)` left the Group A MMCM with a dead input. Fixed by switching to CL-owned MMCM and setting `CLK_GRP_A_EN(0)`. Build `2026_03_19-051231` passes with 0 DRC errors.
